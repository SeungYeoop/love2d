# PROJECT_RULES

## 목적
- 이 프로젝트를 LÖVE2D용 바이브 코딩 템플릿으로 유지한다.
- 짧은 설명만으로도 구조 잡힌 만족스러운 프로토타입을 빠르게 만들 수 있게 한다.
- AI 협업 시 작업 범위를 명확히 유지한다.
- 큰 수정, 과한 추론, 무단 구조 변경을 방지한다.
- 최소 수정 원칙으로 안정적으로 작업한다.

## 기본 작업 루프
1. 먼저 `AI_WORKFLOW.md`, `TASK.md`, 관련 코드와 문서를 읽고 현재 상태를 파악한다.
2. 사용자의 게임 설명과 현재 요청을 3~6줄로 짧게 요약한다.
3. 진행을 막는 정보가 아니면 질문을 줄이고, 합리적인 기본값을 스스로 정한 뒤 바로 구현한다.
4. 첫 구현은 구조가 있는 playable prototype 또는 vertical slice를 우선 목표로 한다.
5. 코드 수정이 끝나면 `.\play.cmd`를 실행하거나 watcher 기준으로 LÖVE2D 반영 여부를 확인한다.
6. 라운드가 끝날 때마다 다음 개선 3개를 추천하거나, 정말 필요한 질문만 1~3개 한다.

## 핵심 규칙
1. 수정 범위가 크거나 여러 파일에 영향이 크면 먼저 계획과 영향 범위를 설명하고 승인받는다.
2. 한 파일에 모든 코드를 몰아넣지 않는다. 기능별로 파일과 모듈을 나눈다.
3. 요청이 모호하더라도 진행 가능한 범위는 먼저 정리하고, 막히는 경우에만 이해한 내용을 2~4줄로 짧게 정리한 뒤 확인받는다.
4. 기존 동작을 함부로 바꾸지 않는다. 동작 변경이 필요하면 이유와 예상 영향을 먼저 설명한다.
5. 항상 최소 수정 원칙을 우선한다.
6. 수정 전에는 어떤 파일을 왜 바꾸는지 설명하고, 수정 후에는 무엇이 바뀌었는지 요약한다.
7. 에러가 나면 바로 고치기 전에 원인 후보와 재현 흐름을 먼저 분석한다.
8. 한 번에 하나의 목적만 수정한다. 버그 수정과 구조 변경을 한 번에 섞지 않는다.

## 프로토타입 품질 기준
- 첫 결과물은 단순 입력 테스트보다 플레이 감각이 있는 상태를 목표로 한다.
- 가능하면 조작법, 목표, 실패/재시작, 점수 또는 진행 피드백을 포함한다.
- 범위를 넓게 벌리기보다 작아도 만족감 있는 핵심 루프를 우선한다.
- 화면, 피드백, UI, 상태 흐름이 필요하면 최소 수준으로라도 포함한다.
- `main.lua`는 가능한 한 얇게 유지하고, 로직이 늘어나면 작은 모듈 단위로 분리한다.

## 작업 방식
- 먼저 현재 구조와 관련 파일을 확인한다.
- 먼저 `TASK.md`의 현재 작업 범위를 기준으로 삼는다.
- 직접 관련된 파일만 수정한다.
- 새 기능은 가능하면 새 파일/모듈로 추가한다.
- 리팩토링, 이동, 이름 변경은 요청 범위 밖이면 하지 않는다.
- 임시 디버그 코드, 콘솔 출력, 사용하지 않는 파일은 남기지 않는다.

## LÖVE2D 구조 원칙
- `main.lua`는 진입점 역할만 유지하는 방향을 우선한다.
- `love.load`, `love.update`, `love.draw`에 모든 로직을 계속 누적하지 않는다.
- 입력, 상태, 엔티티, 렌더링, 씬, 유틸은 분리 가능한 시점에 파일을 나눈다.
- 공통 로직은 재사용 가능한 모듈로 둔다.
- 데이터와 로직, 화면 처리 코드를 한곳에 과도하게 섞지 않는다.
- 전역 변수는 꼭 필요할 때만 사용하고, 가능하면 테이블이나 모듈 안에 묶는다.
- 프레임 업데이트 로직과 그리기 로직을 섞지 않는다.

## 변경 전 체크
- 이 수정이 현재 요청 범위 안에 있는가?
- 기존 동작을 깨지 않고 해결 가능한가?
- 새 파일 추가가 더 안전한가?
- 승인 없이 진행하면 안 되는 큰 수정인가?
- LÖVE 콜백 구조를 불필요하게 바꾸고 있지는 않은가?

## 변경 후 체크
- 요청한 문제만 해결했는가?
- 불필요한 구조 변경이 없는가?
- 다음 AI 세션이 이해할 수 있게 문서가 필요한 만큼만 갱신되었는가?
- `main.lua`가 과하게 비대해지지 않았는가?
- 변경 결과를 LÖVE2D로 다시 확인했는가?
- 다음 개선 제안 또는 필요한 질문을 남겼는가?

## 금지 사항
- 승인 없이 대규모 리팩토링
- 승인 없이 파일 이동/이름 변경
- 관련 없는 코드까지 함께 수정
- 한 파일에 기능 누적
- 추측만으로 요구사항 확정
- 원인 분석 없이 감으로 에러 수정
- LÖVE2D 실행 흐름과 무관한 프레임워크를 멋대로 도입

## 템플릿 메모
- 이 프로젝트는 완성 게임이 아니라 LÖVE2D 작업 시작점으로 사용한다.
- 첫 구현은 작게 시작하되, "플레이 가능한 결과물"까지는 밀고 간다.
- 이후 라운드마다 AI가 개선안을 제안하면서 점진적으로 품질을 높인다.
- 문서는 AI 세션이 바뀌어도 현재 맥락을 빠르게 이어받을 수 있게 유지한다.
- `README.md`는 사용자용 튜토리얼이고, 나머지 운영 문서는 AI 협업용이다.
