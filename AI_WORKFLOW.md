# AI_WORKFLOW

## 목적
- 짧은 게임 설명과 간단한 명령만 받아도 구조 잡힌 LÖVE2D 프로토타입을 빠르게 만든다.
- 첫 결과물은 단순 테스트가 아니라 플레이 가능한 vertical slice를 목표로 한다.
- 각 라운드가 끝날 때마다 AI가 다음 개선 방향까지 이어서 제안한다.

## 기본 제작 루프
1. `PROJECT_RULES.md`, `TASK.md`, `STRUCTURE.md`, `SESSION_SUMMARY.md`, 관련 코드 파일을 먼저 읽는다.
2. 사용자의 게임 설명과 현재 요청을 3~6줄로 요약한다.
3. 막히는 정보가 아니면 질문을 줄이고, 합리적인 기본값을 정한 뒤 바로 구현한다.
4. 첫 구현은 구조 있는 playable prototype으로 만든다.
5. 작업이 끝나면 `.\play.cmd`를 실행하거나 watcher 기준으로 결과가 반영됐는지 확인한다.
6. 마지막에는 다음 개선 3개를 추천하거나, 정말 필요한 질문만 1~3개 남긴다.

## 첫 구현에서 기대하는 품질 바
- 플레이어가 무엇을 해야 하는지 바로 이해된다.
- 조작법이 있다.
- 목표 또는 생존/점수 같은 진행 기준이 있다.
- 실패, 재시작, 다음 시도 흐름 중 최소 하나가 있다.
- 화면 피드백과 게임 감각이 너무 밋밋하지 않다.
- 코드가 커지면 바로 작은 모듈로 나눌 준비가 되어 있다.

## 질문 규칙
- 진행을 막는 정보만 질문한다.
- 질문이 없어도 reasonable default로 갈 수 있으면 바로 구현한다.
- 질문은 1~3개 안으로 줄인다.
- 질문 대신 선택지를 제안할 수 있으면 선택지를 먼저 준다.

## 구현 우선순위
1. 핵심 루프가 분명한지
2. 실제로 플레이 가능한지
3. 구조가 다음 수정에 버틸 수 있는지
4. 피드백과 만족감이 있는지
5. 이후 개선이 자연스럽게 이어지는지

## 프로토타입 우선 체크리스트
- 단순 이동 데모에서 멈추지 않았는가?
- 게임 목표가 있는가?
- 플레이어가 다시 한 판 할 수 있는가?
- 최소한의 UI 또는 안내가 있는가?
- 한 파일에 모든 책임을 밀어 넣고 있지 않은가?

## 라운드 마감 형식
- 이번 라운드에서 구현한 것
- 지금 바로 플레이하는 방법
- 다음 개선 3개
- 필요한 질문이 있으면 1~3개

## 실행 규칙
- watcher가 실행 중이면 파일 저장만으로도 충분하다.
- watcher가 없으면 `.\play.cmd`를 실행한다.
- 자동 실행이 실패하면 실패 이유를 짧게 설명하고 수동 실행 명령을 같이 남긴다.
